# whisperboy ‚Äî main build plan

## Status: üü° PRE-PHASE 0 (skeleton + plans only)

_Repo skeleton, README, CLAUDE.md, and this plan landed in the initial commit. No code yet. Phase 0 is the first phase that actually does anything._

## Stack (locked)

- **Language:** Kotlin
- **UI:** Jetpack Compose
- **Audio:** androidx.media3 (ExoPlayer + MediaSession + **`MediaLibraryService`** ‚Äî see CLAUDE.md for why `MediaLibraryService` rather than `MediaSessionService`)
- **Data:** Room (book / chapter / bookmark cache), DataStore (preferences), SAF / `DocumentFile` (audiobook files ‚Äî no MediaStore, no `READ_EXTERNAL_STORAGE`)
- **Build front-end:** Google's [Android CLI](https://developer.android.com/tools/agents/android-cli) (`android` command, launched April 2026). Wraps Gradle, SDK, install, run. **No Android Studio.**
- **Build back-end:** Gradle (driven via the Android CLI). The repo includes a Gradle wrapper.
- **Unit tests:** Robolectric (JVM-only, zero device).
- **UI tests:** [mobile-mcp](https://github.com/mobile-next/mobile-mcp) over ADB. Current target: headless AVD `medium_phone` (Android 16 / API 36, RSS ~3.2 GB) ‚Äî see Phase 0.
- **Knowledge:** `android docs search` for live Android API guidance. `android-skills-mcp` for the official Android skills inside Claude Code.

## Spiritual sibling

[Voice](https://github.com/PaulWoitaschek/Voice) (GPLv3, by Paul Woitaschek) is the closest functional analog. Whisperboy is **not** a fork ‚Äî written from scratch in Kotlin + Compose, MIT-licensed ‚Äî but Voice's mapped design space is the one we start in:

- SAF-only library, four folder modes (`SingleFile` / `SingleFolder` / `Root` / `Author`)
- Per-book speed, skip-silence, gain stored on the `BookContent` row
- Sleep timer with fade-out, shake-to-resume, end-of-chapter mode, auto-bookmark on fire
- Custom MP4 box parser for M4B chapter markers
- `MediaLibraryService` for Android Auto

When a phase below has a Voice analog, the phase header references the Voice module that solved that problem so a subagent can study prior art before inventing.

## Phase D ‚Äî scanner: walk SAF tree, classify, persist ‚Äî Status: ‚úÖ DONE

Goal: walk every root, classify folders by `FolderType`, materialize Book / Chapter rows in Room, drop dropped books to `inactive` (soft-delete) on rescan. **Voice analog:** `:core:scanner` (`MediaScanner`, `BookParser`, `ChapterParser`, `MediaAnalyzer`).

- [x] **D.1** Room schema baseline: `BookEntity` (book id = SHA-256 of root tree URI + relative path; speed, skipSilence, gain, name, author, cover, currentChapterIndex, positionInChapter, lastPlayedAt, active), `ChapterEntity` (book id, index, name, duration, file uri or position-within-file, list of `ChapterMark`s), `BookmarkEntity` (uuid, book id, chapter id, title, time, addedAt, setBySleepTimer). Schema export on; Room version 1; no `fallbackToDestructiveMigration`. ‚Äî three per-entity files in `data/library/` (R.F.2 ticks): `BookEntity.kt`, `ChapterEntity.kt`, `BookmarkEntity.kt` with FK constraints (`Chapter` ‚Üí `Book` CASCADE, `Bookmark` ‚Üí `Book` CASCADE, `Bookmark` ‚Üí `Chapter` SET_NULL). DAOs land in `BookDao.kt` / `ChapterDao.kt` / `BookmarkDao.kt`, all suspend except observe-Flows (R.F.3 ticks for the data layer). `LibraryDatabase.kt` is `@Database(version = 1, exportSchema = true)`; **no `fallbackToDestructiveMigration`** ‚Äî any v1‚Üív2+ change must ship a migration. Schema JSON written to `app/schemas/com.eight87.whisperboy.data.library.LibraryDatabase/1.json` by Room's gradle plugin (`room { schemaDirectory("$projectDir/schemas") }` in `app/build.gradle.kts`). `ChapterMark` deferred to Phase I (M4B / Matroska parsing); for now chapters are scalar rows. Database constructed in `AppGraph` via `Room.databaseBuilder(... "library.db")` so a misconfigured schema fails fast at app start, not on first scan. Narrow data interfaces shipped alongside (R.A.1 partial tick ‚Äî see `refactor-solid.md`): `BookSource`, `ChapterSource`, `BookmarkSource`. Concrete repository lands in D.4.
- [x] **D.2** `Scanner` walks each root via `CachedDocumentFile`, dispatches by `FolderType`:
  - `SingleFile` ‚Üí one `Book`, chapters from embedded markers (Phase I) or one chapter == whole file
  - `SingleFolder` ‚Üí one `Book`, chapters = audio files in alphabetical order (`BookComparator`)
  - `Root` ‚Üí each first-level subfolder is a separate `Book`
  - `Author` ‚Üí first-level subfolders are authors, second-level are `Book`s

  Shipped as `data/library/SafLibraryScanner.kt` (impl) behind `LibraryScanner` interface (R.A.1 ‚Äî narrow data interface). Dispatch is an exhaustive `when (root.folderType)` over the sealed `FolderType` (R.A pattern: O ‚Äî open/closed; new variants land as new cases, no `if` chains). Scan is a `suspend` function that runs on `Dispatchers.IO` end-to-end (R.F.3); no `runBlocking` on the scan path. Stable identifiers via SHA-256: `bookIdFor(treeUri, relativePath)` and `chapterIdFor(bookId, index)` ‚Äî same input ‚Üí same id across rescans, so per-book settings + bookmarks survive. Audio file detection via `SupportedAudioFormats` (mp3 / m4a / m4b / aac / 3gp / ogg / oga / opus / flac / wav / mka / mkv / webm / mp4 ‚Äî covers Voice's set and the Phase B.5 codec smoke). Output is `ScanSnapshot(books)` with `ScannedBook` / `ScannedChapter` types **separate** from the Room entities (R.F.4 ‚Äî scan-side fields like raw cover bytes can land here without polluting cache schema). Phase D.2 fills structural fields only; `durationMs` / metadata-`title` / `author` / `cover` / `positionInBookMs` stay at defaults until Phase D.3's `MediaAnalyzer` enriches and Phase D.4's `applyScan` computes cumulative offsets. Wired into `AppGraph.libraryScanner` as the narrow interface; concrete impl is internal-able when D.4 lands. Unit tests: `SafLibraryScannerTest` covers ID-stability across calls, ID divergence across treeUri / path / chapter index, and that `bookIdFor` returns 64-char lowercase hex SHA-256. Walking-the-actual-tree behaviour will be covered by D.6's smoke test against a real fixture folder on `emulator-5554`.
- [x] **D.3** `MediaAnalyzer` extracts duration + title/author/cover via `MediaMetadataRetriever` for each audio file. Cover bytes: prefer embedded ‚Üí fallback to `cover.jpg` / `folder.jpg` next to the audio file. ‚Äî shipped as four files in `data/library/`: `MediaAnalyzer.kt` (narrow interface ‚Äî R.A.1 extension ‚Äî plus `FileMetadata` data class with custom `equals`/`hashCode` for the embedded cover `ByteArray`), `Media3MediaAnalyzer.kt` (impl using `android.media.MediaMetadataRetriever`; reads `METADATA_KEY_DURATION` / `_TITLE` / `_ARTIST` (fallback `_ALBUMARTIST`) / `embeddedPicture`; suspend on `Dispatchers.IO`; bad files don't crash the scan ‚Äî `Throwable` caught, returns `null`, retriever released in `finally`), `FolderCoverFinder.kt` (small helper checking `cover.{jpg,jpeg,png,webp}` / `folder.{jpg,jpeg,png,webp}` / `albumart.{jpg,png}` next to the audio, case-insensitive), and `LibraryScannerEnrichment.kt` (glue: walks `ScanSnapshot.books`, calls `MediaAnalyzer.extract` per chapter, fills `durationMs` / `title`, computes cumulative `positionInBookMs` across the chapter list in one pass, rolls book-level fields up ‚Äî `durationMs` = sum of chapter durations, `author` = first non-null per-chapter author). Cover-bytes-to-disk write is deferred to Phase D.4's `applyScan` so the file lifecycle stays bound to the row write transaction. Wired into `AppGraph.mediaAnalyzer` and `AppGraph.libraryScannerEnrichment` as narrow handles. Suspend throughout, no `runBlocking` (R.F.3 still holds). Build + `:app:testDebugUnitTest` clean.
- [x] **D.4** Diff & apply: scanner produces a `ScanSnapshot`; `LibraryRepository.applyScan(snapshot)` writes only changed rows in a single Room transaction, marks gone-but-was-here books `active = false` (soft-delete preserves bookmarks + position if the user re-adds the folder). ‚Äî shipped as four new files in `data/library/`: `ScanWriter.kt` (one-method narrow interface ‚Äî R.A.1 extension ‚Äî separating the write path from the three read sources), `CoverStore.kt` (atomic `<filesDir>/covers/<bookId>` writer; tmp-then-rename, fallback copy-then-delete on rename failure), `LibraryRepository.kt` (`internal class` implementing all four narrow interfaces ‚Äî R.A.2 ticks). Plus extensions: `BookmarkDao.rename(id, title)` for the rename path; `ScannedBook.embeddedCoverBytes` for D.3 ‚Üí D.4 cover-byte propagation, with custom `equals`/`hashCode` on the `ByteArray` field; `LibraryScannerEnrichment` propagates first-non-null embedded cover from per-chapter metadata. **applyScan algorithm:** (1) write any cover bytes to disk *outside* the Room transaction (long file IO inside a txn would hold the database lock; rollback would orphan the file); (2) gather the set of `treeUriString`s touched by the incoming snapshot; (3) `database.withTransaction { ‚Ä¶ }` runs: mark every active book from each touched root inactive (`bookDao.markRootInactive(treeUri)`), then upsert incoming books (which sets `active = true` on the upsert), then for each book delete-and-replace its chapter rows (CASCADE on chapter ‚Üí bookmark FK is `SET_NULL` so bookmarks survive a chapter row disappearing ‚Äî they unpin from the chapter and stay pinned to the book-level position). Result: missing books soft-deleted with bookmarks + per-book settings preserved; added/changed books reflect reality; bookmarks survive chapter reshuffles. AppGraph now exposes `bookSource` / `chapterSource` / `bookmarkSource` / `scanWriter` as narrow handles; the concrete `LibraryRepository` is held privately, never leaked. Per-book settings (resume position / speed / skipSilence / gain / lastPlayedAt) are reset to defaults on rescan in this phase ‚Äî preserving them across rescans is a tracked follow-up that lands when Phase F's playback wires in (it needs the merge-with-existing-row code path). FolderCoverFinder integration (cover.jpg fallback) is also tracked as a follow-up ‚Äî D.3's enrichment doesn't have folder-context yet, so the helper is wired into AppGraph but not called. Build + `:app:testDebugUnitTest` clean.
- [x] **D.5** Rescan triggers: manual (Settings ‚Üí Rescan), on app foreground (debounced 30s), on persistable URI permission added/removed. **No `ContentObserver`** ‚Äî SAF doesn't surface change events the same way `MediaStore` does; we rescan on signal, not on observation. ‚Äî shipped as `data/library/LibraryRescanCoordinator.kt` (narrow interface, R.A.1 extension: `requestRescan()` + `state: StateFlow<RescanState>`) and `AndroidLibraryRescanCoordinator.kt` (impl). ## Phase I ‚Äî chapter parsing (M4B + Matroska) ‚Äî ‚úÖ DONE (shipped in commit 73a2709)

Goal: parse embedded chapter markers from M4B (Apple chap-track + Nero chpl atoms) and Matroska/MKV/WebM (EBML chapters). This is *the* feature that separates a real audiobook player from a music player pointed at long files. Plan it carefully; budget time.

Shipped as a single commit; new package `data/library/parser/` houses every parser plus the seekable-source abstraction so the parsers can be unit-tested on the JVM without `Context`. The MP4 box walker descends 12 container box types (`moov`, `trak`, `mdia`, `minf`, `stbl`, `udta`, `meta` (FullBox), `ilst`, `edts`, `tref`, `moof`, `traf`, `mvex`) capped at depth 16. `LibraryScannerEnrichment` gained an optional `ChapterParser` constructor param + `maybeExpandSingleFileChapters` step that turns a SingleFile book's lone structural chapter into one row per embedded mark. `AppGraph` constructs the parser once and hands it to the enrichment.

- [x] **I.1** `SafSeekableDataSource` ‚Äî random-access reader over a SAF `Uri` (Media3-style `DataSource`-flavored, but for our parser, not for ExoPlayer). Wraps `ContentResolver.openAssetFileDescriptor` and `seek` on the underlying `FileDescriptor`. Necessary because SAF streams are forward-only by default. ‚Äî `FileInputStream.channel` over the `AssetFileDescriptor`'s `FileDescriptor` (RandomAccessFile's `FileDescriptor` constructor is package-private and not callable from Kotlin). Lives behind a `SeekableSource` interface so tests substitute an in-memory `ByteArraySeekableSource`.
- [x] **I.2** `Mp4BoxParser` ‚Äî visitor-based MP4 atom walker. Visit `moov.trak.mdia.minf.stbl.{stco,stsc,stsz,stts}` to build offset tables; visit `moov.udta.chpl` for Nero chapter list; visit `chap` track-reference + corresponding text track for Apple chap-track. ‚Äî visitor returns `Descend`/`Skip`/`Stop`; recursion + container-set guard live in the parser, not the visitor. `meta` payload's 4-byte version+flags skipped before descent.
- [x] **I.3** Apple chap-track decoder ‚Äî for each chunk in the chap text track, decode `(start_time, length, title_string)`. Convert sample-time to seconds via `mdhd` timescale. ‚Äî `Mp4Collector` accumulates per-track `stsz`/`stsc`/`stco|co64`/`stts` tables; `decodeTextSampleTrack` materialises per-sample file offsets + start times from the `mdhd` timescale and reads `[2-byte length][UTF-8 title]` from each sample. UTF-16 BOM handled.
- [x] **I.4** Nero chpl decoder ‚Äî straight `(timestamp_in_100ns, title)` list. Convert to seconds. ‚Äî at `moov/udta/chpl`. Auto-detects the two competing layouts in the wild: 1.0 (`[count-byte][entries]`) vs the widely-deployed mp4v2/Nero variant (`[1-byte reserved][4-byte count][entries]`). Apple chap-track wins when both present.
- [x] **I.5** `MatroskaChapter` parser ‚Äî walk EBML, find `\Chapters\EditionEntry\ChapterAtom`, extract `ChapterTimeStart` (ns) + `ChapterDisplay\ChapString`. Skip ordered / nested-edition variants for v1. ‚Äî variable-length ID + size readers; first edition wins; nested atoms ignored.
- [x] **I.6** Vorbis comment chapter extractor ‚Äî `CHAPTER001`, `CHAPTER001NAME`, etc. Less common but cheap to add. ‚Äî Ogg page reassembler walks pages and packets (segment-table-driven), strips the codec-specific marker (`OpusTags` / `\x03vorbis`), parses the comment block, regex-matches `CHAPTER\d+` / `CHAPTER\d+NAME`. Timestamp parser tolerates `hh:mm:ss.sss`, `mm:ss.sss`, `ss.sss`.
- [x] **I.7** `ChapterParser` dispatch ‚Äî by extension + sniffed magic, route to the right parser. Output: `List<ChapterMark>(positionMs, title)`. Falls back to "one mark at 0 with the file name" if nothing parses. ‚Äî dispatch is MIME ‚Üí extension ‚Üí magic-byte sniff. Exceptions are swallowed (logged under `whisperboy.parser`) so a malformed file never crashes a scan.
- [x] **I.8** Wire into Phase D's scanner ‚Äî `SingleFile` books with parsed chapters get rich `Chapter` rows; without, get one chapter == whole file. ‚Äî `LibraryScannerEnrichment.maybeExpandSingleFileChapters` runs the parser when a book has one structural chapter, expands marks into multi-chapter rows (chapter ids stable via `SafLibraryScanner.chapterIdFor(bookId, index)`), and switches duration computation to position-delta mode so the per-file analyzer doesn't broadcast the full file length to every row.

## Phase P ‚Äî polish + edge cases (the unglamorous one)

Caught-once-then-fixed bugs and gnarly real-world cases that don't belong to any earlier phase. Whisperboy will accumulate these the same way tonearmboy did; this is the catch-bucket so they don't get filed under a wrong phase.

- [ ] **P.1** Cold-start playback resumption ‚Äî `MediaSession.Callback.onPlaybackResumption` returns last book + position. Verified via process-death simulation (`adb shell am force-stop com.eight87.whisperboy` while playing).
- [ ] **P.2** Headset / Bluetooth controls ‚Äî play, pause, prev-chapter (long-press prev), next-chapter (long-press next). Verified on AVD virtual headset + real BT headset.
- [ ] **P.3** Process death during scan ‚Äî partial scan is committed as a Room transaction per-book; killing the process mid-scan loses only the in-flight book.
- [ ] **P.4** SAF permission revoked under us ‚Äî when a tree's `Uri` becomes unreadable, all books from that tree are marked `active = false` (soft-delete) and a banner on the library screen offers to re-pick.
- [ ] **P.5** Very large libraries ‚Äî Voice has #3175 (OOM on huge libraries). Profile whisperboy at 500 books, 10000 chapters; if Room queries on the library screen are blocking, page with `Pager` from `androidx.paging`.
- [ ] **P.6** Foreground service lifecycle on Android 14+ ‚Äî `FOREGROUND_SERVICE_MEDIA_PLAYBACK` strictness around when the service can start in the background. Verify on API 34+ AVD.
- [ ] **P.7** Position-save cadence ‚Äî Voice's "experimental playback persistence" (26.4.3) saves on events not continuously, for battery. Mirror the pattern: save on chapter-end, on pause, on backgrounding, on shutdown ‚Äî not on a 1Hz timer.
- [ ] **P.8** **Skip rescan when nothing changed** (tonearmboy `d0a566d` analog for SAF). Cache the SAF tree fingerprint (document count + max-mtime under each persisted root) in DataStore; on foreground-rescan, compare; if unchanged, no-op. Saves the splash-hold + the scanner pass when the user is just resuming the app with no library deltas.
- [ ] **P.9** **Nav scaffold inset double-consume gotcha** (tonearmboy `5bb2fd2`). When the root scaffold consumes `WindowInsets.statusBars` AND the top app bar also pulls from `LocalWindowInsets`, the status bar gets padded twice ‚Äî visible as a too-tall top inset on edge-to-edge. Audit once at Phase E, audit again at Phase F's full-screen player. Fix is `Modifier.consumeWindowInsets(...)` at the right level.
- [ ] **P.10** **Two-stage Compose splash NOT recommended.** tonearmboy tried it in `f249b0e` to bypass the Android 12+ system circle clip, and reverted in `51d1769` ‚Äî the square overlay was visible for ~half a second before fading and "not worth the moving parts". Whisperboy stays on the standard system splash with the 60% mipmap icon (see [`m3-expressive.md`](m3-expressive.md) gotcha #4). This bullet exists so a future agent doesn't re-discover the same regression.

## Stretch ‚Äî post-1.0 differentiators (planned, not scoped to v1)

These are the differentiators identified in the Voice deconstruction (`docs/plans/sharing-analysis.md` and the README "non-goals" list). **Not v1.** Listed here so they don't fall out of the surface area when v1 ships.

- **S.3** Standalone Wear OS app ‚Äî offline-sync a chapter to the watch, leave the phone behind.

- **S.7** Chapter editor ‚Äî inline rename / split / merge of chapters, write-back as a sidecar `chapters.txt` (no mutation of the original M4B).
